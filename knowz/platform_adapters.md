# Knowz MCP Platform Adapters

This file contains skill templates for integrating Knowz MCP with AI coding platforms. Each skill teaches the platform's AI how to interact with the Knowz MCP server for knowledge management.

Generated by `npx knowz-mcp install`. Skills are placed in platform-specific locations.

## Enterprise Configuration

Before using any endpoints or brand names in these skill templates, check for an `enterprise.json` file in the plugin root directory. If present, use its values:
- `brand` → replaces "Knowz" in all generated content
- `mcp_endpoint` → replaces `https://mcp.knowz.io/mcp`
- `api_endpoint` → replaces `https://api.knowz.io/api/v1`

If absent, use the defaults. When `enterprise.json` is present, omit development endpoint references.

# /knowz-ask — AI-Powered Q&A

Ask questions against your Knowz knowledge vaults.

## Instructions

1. Read `knowz-vaults.md` from the project root for vault configuration and routing rules
   - If not found: proceed without vault scoping, suggest running `/knowz-setup`
2. Parse the question from the user's request
3. **Vault routing:** Match the question against each vault's "when to query" rules
   - One match → scope to that vault
   - Multiple matches → query all matching vaults
   - No match → use the default vault
4. Call `ask_question` with:
   - `question`: the user's question
   - `vaultId`: the matched vault ID (if vault file exists)
   - `researchMode`: `true` for complex questions (multi-part, "how", "why", "compare"), `false` for simple lookups
5. Present the answer naturally, citing the vault source

If MCP tools are not available, report: "Knowz MCP not connected. Run /knowz-setup or set KNOWZ_API_KEY."
```

# /knowz-save — Capture Knowledge

Save an insight, decision, or pattern to a Knowz vault.

## Instructions

1. Read `knowz-vaults.md` from the project root for vault configuration
2. Parse the content from the user's request
3. **Category detection** — scan content for signals:
   - pattern/reusable/utility → Pattern
   - chose/decided/trade-off → Decision
   - workaround/limitation/temporary → Workaround
   - faster/optimized/cache → Performance
   - security/vulnerability/auth → Security
   - always/never/standard/convention → Convention
   - No clear match → Note
4. **Vault routing:** Match content against each vault's "when to save" rules
   - Multiple matches → ask user which vault
   - No match → use default vault
5. **Format content** — expand terse input into detailed, self-contained content:
   ```
   [CONTEXT] Where/why this arose
   [INSIGHT] The knowledge — detailed, actionable
   [RATIONALE] Why this approach, alternatives considered
   [TAGS] category, technology, domain keywords
   ```
6. **Generate title:** `{Category}: {Descriptive summary}`
7. **Dedup check:** Call `search_knowledge` with the title and vault — if similar exists, offer to skip or update
8. Call `create_knowledge` with content, title, knowledgeType "Note", vaultId, tags
9. If MCP write fails, append to `knowz-pending.md` in project root:
   ```
   ---
   ### {timestamp} -- {Category}: {Title}
   - **Category**: {category}
   - **Target Vault**: {vault name}
   - **Source**: knowz-save
   - **Content**: {formatted content}
   ---
   ```
   Report: "Queued to knowz-pending.md — run /knowz-flush when MCP is available."
```

# /knowz-setup — Configure MCP Server

Configure the Knowz MCP server for knowledge management.

## MCP Configuration

**Default endpoint:** `https://mcp.knowz.io/mcp`
**Dev endpoint:** `https://mcp.dev.knowz.io/mcp`

### Step 1: Check MCP Status

Check if Knowz MCP tools (`list_vaults`, `search_knowledge`, etc.) are available.
- If available → skip to Step 3 (vault file creation)
- If not available → proceed with configuration

### Step 2: Configure MCP

**Smart Config Discovery** — before prompting, check:
1. `KNOWZ_API_KEY` environment variable
2. Cross-platform configs: `.gemini/settings.json`, `.vscode/mcp.json`, `.mcp.json`
3. API key from user arguments

If no auth found, ask: API Key or Register (/knowz-register).

**For Codex:** Write `.mcp.json` in project root:
```json
{
  "servers": {
    "knowz": {
      "url": "https://mcp.knowz.io/mcp",
      "headers": {
        "Authorization": "Bearer <api-key>",
        "X-Project-Path": "<project-path>"
      }
    }
  }
}
```

After writing config, the MCP server should be available on next session start.

### Step 3: Vault File Creation

1. Call `list_vaults(includeStats: true)` to discover vaults
2. If `knowz-vaults.md` exists → compare and offer to update
3. If not → present discovered vaults, let user select, generate `knowz-vaults.md`
4. If no vaults on server → offer to create one via `create_vault`
5. If `knowzcode/knowzcode_vaults.md` exists → update vault IDs there too

Report configured vaults and available commands.
```

# /knowz-register — Register & Configure

Create a Knowz account and automatically configure the MCP server.

## Instructions

**Registration API:** `https://api.knowz.io/api/v1/auth/register`
**Dev API:** `https://api.dev.knowz.io/api/v1/auth/register`

### Step 1: Smart Discovery

Check for existing API key in `KNOWZ_API_KEY`, `.gemini/settings.json`, `.mcp.json`.
If found: offer "Use existing key via /knowz-setup" or "Register new account".

### Step 2: Collect Information (one at a time)

1. **Name** — 2-100 characters
2. **Email** — valid email address
3. **Password** — minimum 8 characters (sent securely, never stored locally)

### Step 3: Call Registration API

```bash
curl -X POST https://api.knowz.io/api/v1/auth/register \
  -H "Content-Type: application/json" \
  -d '{"name": "{name}", "email": "{email}", "password": "{password}"}'
```

Extract: API key, vault ID, vault name from response.
Handle errors: 409 (email exists), 400 (validation), 429 (rate limit).

### Step 4: Configure MCP

Write `.mcp.json` with the API key (see /knowz-setup for format).

### Step 5: Generate Vault Configuration

Generate `knowz-vaults.md` with the registered vault.
If `knowzcode/knowzcode_vaults.md` exists, update vault IDs there too.

### Step 6: Success

Report account details, MCP status, and vault configuration.
Note: restart the AI tool session to load the new MCP server.
```

# /knowz-flush — Process Pending Captures

Drain `knowz-pending.md` to vaults.

## Instructions

1. Read `knowz-pending.md` from the project root
   - If not found or empty: report "0 pending captures — nothing to flush." STOP.
2. Verify MCP connectivity — check `create_knowledge` is available
3. Read `knowz-vaults.md` to resolve vault IDs
4. Parse capture blocks (delimited by `---`):
   - Each block: timestamp, title (### header), Category, Target Vault, Source, Content
5. For each block:
   - Resolve target vault ID from `knowz-vaults.md`
   - Call `create_knowledge` with title, content, knowledgeType "Note", vaultId, tags
   - On success: mark for removal
   - On failure: keep in place, log error
6. Update `knowz-pending.md` — remove flushed blocks, keep header
7. Report: flushed count, titles, any failures
```

Triggers when user asks about past decisions, conventions, patterns, or shares learnings worth capturing — e.g. 'why did we...', 'what's our convention for...', 'we decided to...', 'I learned that...'"
---

# Knowz Auto — Frictionless Vault Awareness

Automatically consult vaults or offer to save knowledge when the user's message matches vault rules.

## Prerequisites (ALL must be true)

1. `knowz-vaults.md` exists in the project root
2. MCP tools (`list_vaults`, `search_knowledge`, etc.) are available
3. This is NOT during an explicit /knowz command execution
4. The user's message is NOT a clear code instruction (e.g., "fix this bug", "add a test")

If any prerequisite fails → do nothing.

## Execution

### Step 1: Read `knowz-vaults.md`
Parse each vault's name, ID, "when to query" rules, and "when to save" rules.

### Step 2: Classify the User's Message

**Query signals:** "why did we...", "what's our convention for...", "do we have a standard for...", "check if we've done this before"

**Save signals:** "we decided to...", "I learned that...", "the workaround is...", "going forward, we should...", "save this to vault", "capture this in knowz"

### Step 3: Match Against Vault Rules
- For queries: match against "when to query" rules
- For saves: match against "when to save" rules
- No match → do nothing

### Step 4: Take Action

**For queries:** Search silently, weave findings into response naturally. Don't mention failed searches.

**For saves:** Offer to capture — never auto-save:
1. Identify the insight worth capturing
2. Ask: "This looks like it could be worth saving to {Vault Name}. Want me to capture it?"
3. If agreed: format content, dedup check, create knowledge item
4. If MCP write fails: queue to `knowz-pending.md`
5. If declined: continue normally

## Constraints
- Lightweight only — quick search or save offer, nothing more
- Never auto-save — always ask first
- Never block — if vault lookup fails, continue normally
- Don't announce yourself — include findings naturally
```
